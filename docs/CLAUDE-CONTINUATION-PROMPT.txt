TRADING RULES (ENFORCED IN CODE — DO NOT CHANGE WITHOUT TESTING)
-----------------------------------------------------------------
  • 5-pillar scanner: price $2–$20, rel-vol ≥ 5x, %change ≥ 10%, float < 20M, news
  • Entry signals: micro pullback OR bull flag (A-quality only)
  • Entry gates: MACD > 0, price > VWAP, volume surge ≥ 1.5×, time 7:00–11:00 AM ET
  • Position size: 1% account equity ÷ stop distance
  • Stop: 10 cents per share default, bracket order auto-placed
  • Target: 2× the risk (2:1 R:R minimum)
  • Daily halt: if day P&L ≤ -2% starting equity, bot stops all entries
  • Max trades/day: 10
  • All positions closed by 12:00 PM ET
  • Soft exits: VWAP breach, topping tail, MACD crossover negative

KNOWN ISSUES / THINGS TO VERIFY FIRST

1. The Alpaca WebSocket in data_feed.py uses alpaca-py StockDataStream.
   Verify the correct import path with: python -c "from alpaca.data.live import StockDataStream"
   If it fails, the package may need: pip install alpaca-py --upgrade

2. The bot uses alpaca-py (NOT the legacy alpaca-trade-api).
   Ensure requirements.txt has "alpaca-py>=0.20.0" (not "alpaca-trade-api").

3. The Prisma schema references enums (TradeStatus, BotEventType, etc.).
   Run "npx prisma migrate dev --name init" BEFORE first docker compose up to create migrations.
   Without migrations, the API container will fail to start.

4. pandas-ta may fail to compile on ARM64 (Raspberry Pi OS) with "cannot import ta".
   If so, replace with: pip install ta (a lighter alternative) and update indicators.py:
     from ta.trend import MACD, EMAIndicator
     from ta.volume import VolumeWeightedAveragePrice

5. The scanner's Finviz float scraper may be blocked if too many requests.
   Cache aggressively (already done with self._float_cache) but also add a 1-second delay
   between Finviz calls with: time.sleep(1) in _get_float().

6. PostgreSQL volume (postgres_data) will survive docker compose down but NOT docker compose down -v.
   NEVER run docker compose down -v in production — it deletes all trade history.


ARCHITECTURE DECISIONS (DO NOT CHANGE WITHOUT READING THIS)
-------------------------------------------------------------
1. Bracket orders: The bot uses Alpaca bracket orders (entry + stop + target in ONE call).
   This ensures the stop is ALWAYS in the market — even if the bot crashes after entry.
   DO NOT change to separate stop orders — this is a safety feature.

2. Fire-and-forget telemetry: All telemetry.py calls use background threads.
   The trading loop must NEVER be blocked by network I/O to the API.
   DO NOT make telemetry calls synchronous.

3. A-quality signals only: The bot only trades confidence="A" signals.
   B-quality signals are logged to telemetry for analysis but never executed.
   DO NOT lower this gate — it will increase losses.

4. Settings.yaml is the single source of truth for all parameters.
   The config is volume-mounted into the container so changes take effect on next run.
   DO NOT hardcode values that belong in settings.yaml.

5. All times are Eastern Time (ET). Docker TZ=America/New_York.
   APScheduler is initialized with ZoneInfo("America/New_York").
   DO NOT use UTC times in the scheduler.


TESTING APPROACH
-----------------
Before running on real paper account:
  1. Unit test indicators: python -m pytest bot/tests/ (create tests/ folder)
  2. Dry run: comment out broker.place_bracket_order() call in main.py,
     let the bot log signals without placing orders for 1 full day
  3. Check telemetry: open dashboard, verify signals/scans appear in real time
  4. Paper trade: run with TRADING_MODE=paper for minimum 2 weeks
  5. Validate: 100+ trades, positive expectancy (accuracy × avg_winner > avg_loser)
  6. ONLY THEN set TRADING_MODE=live with minimum position sizes
